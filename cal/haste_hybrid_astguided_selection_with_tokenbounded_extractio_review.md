# Harsh Critic Review
## Section-by-Section Critical Review

### Title & Abstract
- **Does the title accurately reflect the contribution?** Yes. `HASTE: Hybrid AST-guided Selection with Token-bounded Extraction` clearly captures the core components of the proposed pipeline.
- **Does the abstract clearly state the problem, method, and key results?** The problem (context window bottleneck forcing a trade-off between structural integrity and semantic relevance) is well-stated. The method (integrating AST constraints, hybrid IR, and budget-aware selection) is summarized. However, the key results are stated at a high level (`up to 85% code compression`, `significantly improving the success rate`) without specifying compared baselines or statistical confidence.
- **Are any claims in the abstract unsupported by the paper?** Yes. The claim `reducing model-generated hallucinations` appears in the abstract and Section 1, and a `Hallucination Rate` metric is formally defined in Section 4.2.3, **but no numerical results for this metric are reported in Section 5**. The paper only reports Judge Scores and Compression Ratios, leaving this central claim empirically unsupported.

### Introduction & Motivation
- **Is the problem well-motivated? Is the gap in prior work clearly identified?** The motivation is clear and practically relevant for SE/LLM research. The dichotomy between structure-aware (AST-heavy) and relevance-focused (IR-heavy) approaches is well-articulated.
- **Are the contributions clearly stated and accurate?** The three contributions are clearly listed. However, Contribution 3 claims an analysis of the compression-quality trade-off. As discussed below, this analysis relies on only six data points, which overstates the robustness of the finding.
- **Does the introduction over-claim or under-sell?** It slightly over-claims generalizability. Phrases like `robust LLM-as-a-judge framework` and `significantly improving the success rate` imply rigorous, large-scale validation, which the current experimental section does not substantiate. The introduction under-specifies how HASTE differs algorithmically from modern code RAG pipelines that already use AST chunking + hybrid retrieval.

### Method / Approach
- **Is the method clearly described and reproducible?** Not currently for ICLR standards. Sections 3.1–3.4 describe the modular pipeline at a systems-engineering level, but lack algorithmic precision for the core novelty. Specifically, Section 3.3 states the expanded call graph set `is then filtered under a strict token budget`, and Section 2.2 claims `AST-bounded pruning... guarantees parent-child relationships remain intact`. The paper does not specify **how** nodes are prioritized, merged, or pruned when the token limit is approached. Is it a greedy subtree selection? Does it prune leaves vs. internal nodes? How are dependency conflicts resolved? Without this, the method cannot be reproduced.
- **Are key assumptions stated and justified?** The pipeline assumes static AST parsing and static call-graph construction are reliably feasible. For Python (used in the experiments), dynamic dispatch, reflection, and `import *` patterns routinely break static call graphs. This assumption is stated implicitly but not justified or addressed as a failure mode.
- **Are there logical gaps in the derivation or reasoning?** The RRF formula in Section 3.3 is standard IR practice, but the paper does not explain how BM25 scores and dense semantic scores are calibrated or weighted before fusion, beyond stating `k=60`. The mapping from retrieval scores to AST-node priority during budget-constrained selection is logically missing.
- **Are there edge cases or failure modes not discussed?** Yes. Files with deep nesting, macros/templating (not applicable to Python but general SE concern), or highly polymorphic codebases are not discussed regarding how the AST budgeting behaves. Circular call graph dependencies are also unaddressed.

### Experiments & Results
- **Do the experiments actually test the paper's claims?** Partially. They test compression and LLM-as-a-judge quality, but fail to report results for `AST Fidelity` and `Hallucination Rate` (Section 4.2.2/4.2.3). The SWE-PolyBench evaluation (Section 5.3) shows a bimodal distribution heavily skewed by trivial "no-op" tasks (score 100), which does not rigorously test automated code editing success.
- **Are baselines appropriate and fairly compared?** No. Baselines are `IR-only`, `AST-only`, and `Naive truncation`. These are overly simplistic for ICLR. Modern code-context methods (e.g., RepoBench, ASTRA, CodeRAG, RepoCoder, or recent context-window optimization papers) are completely omitted. Furthermore, **Table 2 only reports HASTE's scores**. Without baseline scores in the same table or figure, claims of `significant improvement` are unverifiable.
- **Are there missing ablations that would materially change conclusions?** Yes, critically. There is no ablation to isolate components: (1) Does hybrid RRF meaningfully outperform pure semantic retrieval for code? (2) How much does call-graph expansion contribute vs. AST bounding? (3) What is the performance degradation when removing the token budget constraint vs. aggressively applying it? Without these, the necessity of the specific `HASTE` integration is unclear.
- **Are error bars / statistical significance reported?** No. Section 4.1.4 mentions `each task was executed three times and averaged`, but no standard deviation, confidence intervals, or paired statistical tests are reported anywhere in Table 2 or Figures 2/3.
- **Do the results support the claims made, or are they cherry-picked?** The Pearson correlation `r = -0.97` (Section 5.2) is calculated on exactly **N=6** data points. Making a strong statistical claim based on six observations is statistically invalid and misleading. The heavy reliance on trivial "no-op" tasks in SWE-PolyBench (7 out of ~11 reported instances) also appears to cherry-pick easy cases to inflate average scores.
- **Are datasets and evaluation metrics appropriate?** The curated dataset (6 Python files) is far too small for a robust benchmark (Section 4.1.1). SWE-PolyBench is appropriate, but excluding instances that resulted in `processing errors` without reporting how many or what caused them introduces survivorship bias. LLM-as-a-judge is used as the primary metric but is highly susceptible to prompt bias and stylistic preference; without a ground-truth metric (e.g., exact match to reference patch, or automated unit test execution pass@k), the evaluation lacks objective rigor expected at ICLR.

### Writing & Clarity
- **Are there sections that are confusing or poorly explained?** Section 3.4 (Observability) and Section 3.2 (Index Builder) read like generic system architecture documentation rather than a machine learning/contributions paper. They describe what a database does (`persisted in a manner that supports both scalability`) rather than what is novel or relevant to the scientific contribution. Section 5.3's textual description of Figure 3 contradicts the narrative of robust performance; clarifying that half the meaningful tasks failed or scored ≤50 would be more transparent.
- **Are figures and tables clear and informative?** Figure 2(c) and 2(d) show correlations, but as noted, the sample size undermines them. Table 1 and Table 2 are structurally clear. Missing baseline comparisons in Table 2 significantly reduces its informational value.

### Limitations & Broader Impact
- **Do the authors acknowledge the key limitations?** The authors limit their discussion to future work in the Conclusion (cross-file analysis, multi-language support, additional ranking signals). They do not acknowledge the fundamental limitations of their current study: small N, lack of objective execution-based metrics, absence of modern baselines, and the brittleness of static call graphs in Python.
- **Are there fundamental limitations they missed?** Yes. The token-budgeting mechanism's behavior when a critical dependency (e.g., a base class) pushes the context over budget is unaddressed. The system's computational latency for AST parsing, indexing, and graph traversal in large repos is not measured, though Section 3.4 mentions `Latency` as a collected metric. Without reporting it, readers cannot assess deployment feasibility.
- **Are there failure modes or negative societal impacts not discussed?** The failure modes on SWE-PolyBench (Section 5.3) are described, but the paper does not discuss the safety implications of providing structurally coherent but semantically misleading context (e.g., HASTE compresses away a security check, the LLM generates a "safe" edit that compiles but is vulnerable). This is a relevant broader impact for AI-assisted SE.

### Overall Assessment
The paper addresses a highly relevant problem in AI-assisted software engineering: balancing semantic relevance and structural coherence under strict context budgets. The motivation is clear, and the proposed integration of AST constraints with hybrid retrieval is practically sensible. However, the paper currently **does not meet the ICLR bar for empirical rigor and technical specificity**. The core algorithmic contribution—the exact mechanism for AST-guided, token-bounded pruning—is described abstractly without reproducible algorithmic detail. The experimental evaluation relies on a tiny dataset (N=6), lacks comparison against modern baselines and ablation studies, and makes statistically unsupported claims (e.g., correlation coefficients on six points). Crucially, metrics defined in Section 4.2 (`AST Fidelity`, `Hallucination Rate`) are never reported in Section 5, leaving key claims about hallucination reduction and structural integrity empirically hollow. To be competitive for ICLR, the authors must (1) formalize the budget-constrained selection algorithm pseudocode, (2) report comprehensive baseline comparisons and ablation studies, (3) provide statistically sound results with error bars on a substantially larger benchmark, (4) report the results for all defined metrics including hallucination rates, and (5) ground evaluation in execution-based correctness (e.g., pass@k on unit tests) rather than relying solely on LLM-as-a-judge.

# Neutral Reviewer
## Balanced Review

### Summary
This paper introduces HASTE, a modular framework designed to resolve the trade-off between semantic relevance and structural coherence when retrieving code context for LLMs. By combining AST-aware chunking, hybrid lexical-semantic retrieval (BM25 + embeddings with Reciprocal Rank Fusion), call-graph expansion, and strict token-bounded pruning, the system aims to deliver compact, compilable contexts that improve downstream code-editing success. Experiments on a small curated dataset and the SWE-PolyBench benchmark report high LLM-as-judge scores and substantial compression rates, arguing that structural preservation mitigates hallucination and supports reliable automated edits.

### Strengths
1. **Clear Problem Formulation & Motivation**: The paper accurately identifies the "structure vs. relevance" dilemma in code context window engineering and frames the design choices effectively around preserving syntactic validity while maintaining semantic recall.
2. **Well-Structured, Modular Pipeline**: The architecture description (Sec 3) logically flows from ingestion and AST-aware chunking to hybrid indexing, retrieval, and budget-constrained selection. The system design is practical, with explicit consideration for token budgets, call-graph expansion, and observability.
3. **Practical Deployment Focus**: The framework is presented as an open-source, pip-installable tool with built-in metrics and logging. This engineering maturity suggests real-world applicability and lowers the barrier for adoption by practitioners.

### Weaknesses
1. **Statistically Thin & Incomplete Evaluation**: The compression-quality correlation relies on only 6 data points (Pearson’s r = -0.97 reported for n=6), which provides no meaningful statistical confidence. Furthermore, Sec 4.2 introduces *AST Fidelity* and *Hallucination Rate* as core metrics, but Sec 5 reports **zero results** for them, leaving the paper's hallucination-mitigation claim unsubstantiated.
2. **Insufficient Baselines & Lack of Ablations**: Comparisons are limited to naive BM25, AST-only traversal, and naive truncation. The paper omits modern code-RAG and context-compression baselines (e.g., RepoCoder, CodeRAG, REPORAINT, or token-pruning methods like CodePrune). There are also no ablation studies quantifying the marginal contribution of RRF, call-graph depth, or the AST-bounded pruning step.
3. **Heavy Reliance on Unvalidated LLM-as-a-Judge**: All primary performance claims rest on a single proprietary LLM (Gemini 1.5 Flash) acting as judge. There is no inter-annotator agreement check, prompt calibration, human validation, or discussion of known judge biases, which weakens the reliability of the reported 90–100 scores.
4. **Missing Experimental Details**: Key hyperparameters (exact token budget values, RRF smoothing choices beyond k=60, maximum call-graph depth, chunking granularity, and the exact judge/prompt templates) are omitted, hindering rigorous reproduction and fair comparison.

### Novelty & Significance
* **Novelty**: Moderate to low from a machine learning perspective. The individual components (AST parsing, hybrid BM25+vector indexing, RRF fusion, call-graph expansion, and budget-aware truncation) are well-established in information retrieval and program analysis. HASTE’s novelty lies primarily in the engineering integration of these techniques into a unified pipeline rather than in algorithmic or learning-driven advances. For ICLR, which typically prioritizes novel learning paradigms, architectural innovations, or rigorous empirical ML contributions, the method reads more like a robust software engineering system than a machine learning breakthrough.
* **Clarity**: High. The paper is well-organized, motivations are clear, and the pipeline is described with appropriate technical depth. Figure placement and table formatting artifacts are clearly parser issues and do not obscure the scientific narrative.
* **Reproducibility**: Moderate. The authors commit to open-sourcing the code and claim PyPI availability, which is positive. However, reliance on a proprietary LLM for both generation and evaluation, combined with omitted prompts, budget thresholds, and graph-expansion hyperparameters, limits full independent reproduction. 
* **Significance**: High for applied AI for Software Engineering, but limited for core ICLR research. The framework addresses a genuine bottleneck in LLM-assisted development. However, without rigorous ablations, broader benchmarks, and validated evaluation metrics, the scientific impact on the broader ML community remains constrained. The work would likely find a stronger reception at a software engineering or applied AI venue (e.g., ICSE, FSE, ESEC/FSE).

### Suggestions for Improvement
1. **Report All Promised Metrics & Validate the Judge**: Provide empirical results for AST Fidelity and Hallucination Rate. To strengthen credibility, supplement or cross-validate the LLM-as-a-judge with human annotations, calculate inter-rater agreement, and release the exact scoring rubrics and prompts.
2. **Expand Baselines & Conduct Ablations**: Compare against state-of-the-art code-RAG and context-pruning methods. Add systematic ablations (e.g., w/o AST chunking, w/o call-graph, w/o RRF, varying token budgets) to isolate the contribution of each component and prove the claimed synergy.
3. **Enlarge & Diversify the Evaluation Dataset**: Move beyond 6 curated files. Use larger, standardized code-reasoning benchmarks (e.g., SWE-bench, CrossCodeEval, CodeRAG-Bench) and perform proper statistical significance testing (e.g., paired bootstrap or Wilcoxon tests) rather than reporting Pearson correlation on minimal samples.
4. **Detail Experimental Hyperparameters & Scope**: Explicitly state token budget sizes, maximum call-graph depth, embedding model choices, chunking rules, and the exact prompts used for both code generation and judging. Clarify how the system handles multi-file dependencies and cross-repo scope, which are critical for real-world SE tasks.
5. **Reframe ICLR Alignment or Target Appropriately**: If submitting to ICLR, emphasize any learning-based components, robust generalization across languages/models, or methodological insights on retrieval-compression trade-offs that transfer beyond software engineering. Otherwise, consider targeting a premier AI-for-SE or systems venue where engineering integration and tooling maturity are highly valued.

# Spark Finder Review
## How to Improve This Paper

### Missing Experiments (top 3-5 only)
1. Compare against state-of-the-art code RAG and context-pruning baselines (e.g., RepoCoder, CodeRAG, or LCC) instead of only naive BM25, raw AST traversal, and truncation. Without this, the claim that HASTE "resolves the relevance vs. structure trade-off" is unverified against the current research frontier and lacks competitive grounding.
2. Include a rigorous component ablation study that independently disables call-graph expansion, hybrid RRF fusion, and the AST-bound token constraints. Without quantifying each module's marginal impact, it is impossible to attribute the observed gains to the proposed hybrid architecture rather than a single strong component or the chosen embedding model.
3. Evaluate task success using executable metrics (e.g., test pass@k or compilation success rate) across a full, unfiltered benchmark suite like SWE-bench Lite. Relying on a hand-picked corpus of six files and a filtered SWE-PolyBench subset invites severe selection bias and fails ICLR’s standard for generalizable, statistically sound empirical evaluation.

### Deeper Analysis Needed (top 3-5 only)
1. Replace the six-point correlation analysis with a large-scale sensitivity study across dozens of repositories. A Pearson r = -0.97 calculated on six data points is statistically meaningless; without broader validation, the claim that HASTE "effectively navigates the compression-quality frontier" is mathematically unsupported.
2. Quantitatively report the defined Hallucination Rate against all baselines and provide error breakdowns by hallucination type. The abstract and contributions explicitly claim reduced LLM hallucinations, but the results section completely omits this metric’s numerical values, rendering the core hallucination-reduction claim purely speculative.
3. Provide a precise computational definition and implementation details for AST Fidelity, then report it across conditions. Without a concrete, reproducible metric showing that AST-guided pruning actually preserves syntactic dependencies better than token pruners, the warning against "Frankenstein contexts" remains a theoretical assertion rather than an empirical finding.

### Visualizations & Case Studies
1. Provide side-by-side code context visualizations showing exactly which lines/functions HASTE kept versus IR-only and AST-only baselines for representative high- and low-scoring tasks. This directly reveals whether the hybrid mechanism correctly isolates critical dependencies or indiscriminately prunes necessary logic.
2. Plot a token-budget sensitivity curve showing downstream task success rate as the maximum context window systematically shrinks. This is required to visually prove the system’s claimed robustness under strict token constraints and to define the operational limits where the method actually begins to fail.
3. Map SWE-PolyBench failure modes to specific retrieval gaps (e.g., missing class definition, unexpanded cross-function dependency, or wrong lexical match). Case studies linking errors to broken structural links would prove whether failures stem from the retrieval pipeline’s blind spots or inherent LLM reasoning limits.

### Obvious Next Steps
1. Replace the single proprietary LLM-as-a-judge with standardized, deterministic evaluation protocols or multiple open-source judges with inter-rater agreement reporting. Sole reliance on one closed-source judge introduces unquantifiable bias and violates ICLR's reproducibility expectations for automated evaluation.
2. Release complete prompt templates, embedding model versions, RRF hyperparameters, and baseline implementation details alongside the code. The methodology lacks the precise configuration specifications required to reproduce the hybrid indexing and token-bounding pipeline, severely undermining experimental credibility.
3. Extend the pipeline to cross-file, repository-level retrieval tasks with explicit dependency tracking. Claiming HASTE is a "step towards scalable AI-assisted software development" requires validation on multi-file codebases where real-world context bottlenecks and structural dependencies actually exist.

# Final Consolidated Review
## Summary
This paper introduces HASTE, a modular pipeline for retrieving code context under strict LLM token budgets. By combining AST-aware chunking, hybrid lexical-semantic retrieval (BM25 + dense embeddings via RRF), call-graph expansion, and token-bounded pruning, the system aims to deliver compact yet structurally coherent code snippets. The framework is evaluated on a hand-curated set of six Python files and a filtered subset of the SWE-PolyBench, reporting high compression ratios and strong LLM-as-judge scores for localized code editing tasks.

## Strengths
- **Precise Problem Formulation:** Clearly identifies a critical, under-appreciated bottleneck in AI-assisted development: naive semantic retrieval often severs syntactic dependencies, producing "Frankenstein contexts" that trigger LLM failures. Framing retrieval around *compilability-aware* context bounds is practically sound.
- **Cohesive Engineering Integration:** The pipeline logically unifies well-established program analysis and IR techniques (AST parsing, hybrid indexing, graph traversal, budget-aware filtering) into a deployable, observation-friendly system. The emphasis on token constraints reflects a realistic engineering constraint rather than an abstract optimization goal.

## Weaknesses
- **Central Claims Are Empirically Unsubstantiated:** The abstract and introduction explicitly claim that HASTE reduces model hallucinations, and Section 4.2 formally defines *Hallucination Rate* and *AST Fidelity* as primary evaluation metrics. However, Section 5 reports **zero numerical results** for these metrics. Without them, the paper's core contribution remains rhetorically asserted rather than empirically validated.
- **Statistically Invalid Analysis & Severe Evaluation Bias:** The compression-quality trade-off analysis calculates a Pearson correlation of *r = -0.97* on exactly **N = 6** data points, which is statistically meaningless. Furthermore, the SWE-PolyBench evaluation explicitly excludes instances with "processing errors" without quantifying them, and heavily features trivial "no-op" tasks. This survivorship and difficulty bias artificially inflates perceived performance and obscures how the system handles complex, multi-file reasoning.
- **Algorithmic Opacity & Lack of Competitive Grounding:** The specific mechanism for *token-bounded AST pruning* is described abstractly. The paper never specifies how candidate nodes or subtrees are prioritized, merged, or truncated when the context budget is exceeded (e.g., greedy leaf pruning vs. dependency-critical retention), making reproduction and technical assessment impossible. Additionally, baselines are restricted to naive truncation, raw BM25, and pure AST traversal, omitting modern code-RAG, context-pruning, and repository-scoped retrieval methods. The sole reliance on a single proprietary LLM-as-a-judge, without execution-based validation (e.g., unit test pass@k), fails to meet ICLR's rigor standards for code generation tasks.

## Nice-to-Haves
- Provide computational overhead/latency metrics for AST parsing, indexing, and graph expansion to quantify the retrieval pipeline's feasibility in real-time developer workflows.
- Extend the pipeline to cross-file and repository-level dependency tracking, as production-scale software engineering rarely confines to single-file edits.

## Novel Insights
The paper empirically reinforces a paradigm shift crucial for applying RAG to code: *compilability-aware retrieval outperforms pure semantic matching*. By demonstrating that aggressive token reduction can coexist with high downstream task quality when bounded by program syntax trees, the work highlights that code context engineering must prioritize structural executability over lexical recall. This finding directly challenges the default transfer of natural-language RAG pipelines to software engineering, where semantic relevance without syntactic integrity actively degrades LLM reasoning.

## Potentially Missed Related Work
- *CodeRAG / RepoBench / RepoCoder:* Modern repository-aware retrieval and code generation frameworks that already explore multi-file graph construction, AST chunking, and iterative context refinement. Comparing against these would better isolate HASTE's unique contribution in budget-constrained pruning.
- *LLM Context Compression Methods (e.g., LongLLMLingua, Selective Context):* Techniques designed for token budget management in LLM prompts. Benchmarking against these would clarify whether AST bounding provides gains beyond standard linguistic compression heuristics.

## Suggestions
- **Report all formally defined metrics** (Hallucination Rate, AST Fidelity) across all conditions, and supplement the N=6 analysis with a statistically robust evaluation on a standardized, execution-verified benchmark (e.g., SWE-bench Lite or CrossCodeEval) to demonstrate generalization.
- **Provide algorithmic pseudocode and hyperparameters** for the token-bounded AST selection step. Clearly define the node/subtree prioritization heuristic, maximum call-graph depth, and exact RRF weighting/k values to ensure reproducibility.
