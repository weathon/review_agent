# Harsh Critic Review
## Section-by-Section Critical Review

### Title & Abstract
The title accurately reflects the scope, but the abstract contains two claims that require tempering or clarification:
1. **"manually validated to eliminate ill-defined problems"**: Section 3.1 clarifies that only the 494 *base instances* undergo manual inspection. The 1,794 total tasks are largely synthetically augmented. The abstract should distinguish between manual validation of base instances and automated generation of derived tasks, as the latter introduces its own quality risks (e.g., invalid perturbed patches or noisy linter outputs).
2. **"avoid data leakage issues"**: OmniCode relies heavily on repositories from SWE-Bench and Multi-SWE-Bench. Given that frontier models are trained on massive corpora containing these popular open-source projects, synthetic task generation (e.g., creating bad patches or linting violations) does not obscure the underlying code context. The authors should clarify what leakage means in this context and provide evidence or reasoning why their synthetic pipeline meaningfully mitigates it, rather than just shifting the task framing.

### Introduction & Motivation
The motivation for broadening evaluation beyond bug-fixing is strong and well-aligned with current needs in agent research. However, the selection of the four task categories (Issue Resolution, Test Generation, Code Review, Style Fixing) appears somewhat arbitrary. Why these four over other high-impact SE tasks like refactoring, debugging/trace analysis, or dependency management? A clearer principled justification (e.g., covering the full PR lifecycle, or targeting distinct reasoning capabilities like verification vs. generation vs. style adherence) would strengthen the contribution. Additionally, the introduction claims the benchmark is "not intractable" while showing maximum test-gen success at 25%. This is a useful finding, but the intro should more carefully frame what performance levels should be targeted given the task difficulty.

### Benchmark Construction (Method)
This section is the core of the paper but raises several methodological concerns that need addressing before the benchmark can be considered robust.

* **Base Data & Reproducibility (Sec 3.1)**: Only 37 out of 494 instances are newly curated; the rest are drawn from existing benchmarks. While this accelerates development, it heavily ties OmniCode's distribution to prior curation biases. Furthermore, containerized environments are mentioned, but dependency resolution for multi-language, multi-framework repositories is notoriously brittle. The paper must explicitly state whether these environments and build scripts will be publicly released, as this is a hard requirement for ICLR reproducibility standards.
* **Test Generation Metric (Sec 3.2.2)**: The requirement that a generated test must pass the gold patch *and fail all bad patches* is extremely strict. If any single bad patch is functionally equivalent to the gold patch on the specific inputs covered by the test, the agent will be unfairly penalized. Additionally, bad patches are generated using `Agentless` (specific models listed) and perturbation prompts. This creates a **distributional coupling**: the benchmark's difficulty and discriminative power are intrinsically tied to the failure modes of specific, now-dated, LLMs. If future agents solve problems via fundamentally different pathways that don't resemble `Agentless`'s errors, the "bad patches" become irrelevant distractors. The authors should discuss this limitation and consider how to validate or diversify bad patches independently of specific agent failures.
* **Code Review Tasks (Sec 3.2.3)**: Reviews are LLM-generated from LLM-generated bad patches. The quality of a review instruction directly dictates task feasibility. The paper lacks human validation of review informativeness. If a review is vague or hallucinates constraints, the agent's failure may reflect prompt noise rather than poor coding ability. Appendix A categorizes reviews using an LLM, which risks circular validation. At minimum, a sample of reviews should be manually audited, or the prompting strategy should include strict constraints verified by a separate verifier.
* **Style Fixing Metric & Tool Dependency (Sec 3.2.4 & Eq. 1)**: Style linting is highly tool-config-specific. The paper mentions "aggressively pruning overly zealous rules" but does not list which rules were removed or provide the exact tool configurations. This is a critical reproducibility gap. Furthermore, the scoring formula `max((resolved - new) / original, 0)` has counterintuitive properties: it treats fixing one critical style violation identically to fixing three minor ones, and it heavily penalizes agents that attempt broad refactors that inadvertently trigger new linter warnings. Style tools also produce false positives; penalizing agents for tool noise conflates linter limitations with agent capability.

### Experiments & Results
* **Statistical Rigor**: All results are presented as single-point estimates. LLM outputs are highly stochastic, especially for repository-level agentic workflows. ICLR standards strongly expect variance reporting (e.g., multiple seeds, confidence intervals, or bootstrapped error bars). Without this, observed differences (e.g., 14.0% vs 9.4% in Table 3) cannot be distinguished from random noise.
* **Baseline Comparisons**: The paper compares `SWE-Agent` vs. `Aider`, which is a useful agentic vs. pipeline comparison. However, there is **no ablation or zero-shot baseline**. A simple prompt-to-patch baseline (without agentic loops, file editing tools, or execution feedback) is missing. It's unclear how much performance actually comes from the agent framework vs. the underlying LLM's raw coding ability. Adding a pass@k code generation baseline or a minimal ReAct agent would significantly strengthen the analysis.
* **Review-Response Analysis (Sec 5.3)**: The hypothesis that structured feedback improves performance is interesting, but the analysis is confounded. The authors note that Python Bug-Fix (56.4%) outperforms Review-Response (52.2%), then explain this by claiming the non-review subset is easier. This hand-wavy subset explanation weakens the claim. A controlled, matched-pair comparison (evaluating the exact same instances under both framings) with statistical testing is required to substantiate the hypothesis.
* **Missing Data in Tables**: Table 4 shows `--%` for C++ and Java under certain experimental settings. These missing values need clarification and imputation/explanation to avoid appearing as cherry-picked reporting.

### Limitations & Broader Impact
The limitations section is brief and focuses mostly on future task expansion. It misses several critical methodological limitations explicitly identified above:
1. The synthetic bad-patch generation pipeline inherits biases from specific base models, potentially dating the benchmark's difficulty.
2. LLM-generated reviews and linter-driven style evaluation introduce automated noise that may conflate tool limitations with agent failures.
3. Strict evaluation metrics (failing all bad patches) may undercount useful tests that are semantically valid but fail due to trivial bad-patch overlaps.
4. The fixed $2.0 instance cost limit likely caps performance on complex tasks; this should be acknowledged as a constraint on measured capability, not an absolute capability ceiling.

## Overall Assessment
OmniCode addresses a genuine and important gap in software engineering benchmarks by moving beyond isolated bug-fixing into test generation, code review, and style enforcement. The multi-language scope and emphasis on repository-level evaluation are highly relevant to ICLR's interests in agent capabilities and real-world applicability. However, the paper currently falls short of ICLR's rigor standards in three key areas: (1) **Evaluation Design**: The test-generation metric is overly strict and coupled to specific LLM failure modes, while style evaluation relies on opaque tool configurations; (2) **Experimental Rigor**: Results lack statistical variance reporting, and the absence of non-agentic baselines makes it difficult to isolate framework vs. model contributions; (3) **Reproducibility & Leakage Claims**: Heavy reliance on known repositories without clear mitigation for code-context leakage, combined with missing prompt details in Appendix B, raises reproducibility concerns. With clarified methodology, controlled matched-pair analyses, statistical reporting, and a more nuanced discussion of synthetic data limitations, this could be a strong benchmark contribution. As submitted, it requires substantial methodological clarification and experimental strengthening.

# Neutral Reviewer
## Balanced Review

### Summary
OmniCode introduces a multi-task, multi-language benchmark (1,794 instances across Python, Java, and C++) designed to evaluate LLM-powered software engineering agents beyond standard issue resolution. By synthetically deriving tasks for test generation, code review response, and style enforcement from manually curated GitHub pull requests, the authors establish a rigorous evaluation pipeline that incorporates "bad patches" to prevent superficial test passing. Empirical evaluations reveal significant performance disparities across tasks and languages, highlighting agent struggles with robust test generation and complex iterative reasoning.

### Strengths
1. **Rigorous, Realistic Evaluation Design:** The benchmark moves beyond simple pass/fail on golden patches by introducing "bad patches" for the test generation task. Requiring generated tests to pass the correct fix while failing plausible incorrect fixes (Sections 3.2.2 & 5.5) significantly raises the evaluation bar and better reflects real-world testing standards.
2. **Comprehensive Task & Language Coverage:** Expanding evaluation to four distinct workflow stages (bug fixing, test generation, review response, style fixing) across three major languages addresses a critical gap in Python-centric benchmarks like SWE-Bench. The integration of language-specific static analysis tools (pylint, clang-tidy, PMD) ensures cross-linguistic consistency in the style task (Section 3.2.4).
3. **Insightful Diagnostic Analysis:** The authors provide valuable behavioral insights beyond aggregate scores, such as the strong correlation between bug-fixing and review-response performance (Section 5.1), the "explosive" patch complexity observed in unresolved instances (Section 5.4), and meaningful differences between agentic (SWE-Agent) and pipeline-based (Aider) architectures (Section 5.2).

### Weaknesses
1. **LLM-Reliant Review Generation Lacks Human Validation:** The code review task relies entirely on a single LLM (Gemini 2.0 Flash) to generate feedback without human quality auditing (Section 3.2.3). This risks propagating hallucinated, unhelpful, or stylistically biased reviews that could mislead or unfairly penalize agents, weakening the validity of this task category.
2. **Limited Baseline Coverage & Ambiguous Model Naming:** The evaluation compares only SWE-Agent and Aider. ICLR benchmark submissions typically require broader comparisons against contemporary frameworks (e.g., OpenDevin, CodeR, or specialized test generators) to contextualize results. Additionally, references to "GPT5-mini" and "DeepSeek-V3.1" appear to be placeholders or typographical errors, which hinders immediate reproducibility.
3. **Unsubstantiated Style Metric & Thresholding:** The custom style-fixing formula (`(original - resolved - new) / original`, Section 3.2.4) is mathematically underspecified and lacks justification against standard precision/recall/F1 metrics for linting. Without clear per-language threshold explanations or analysis of how different linters weight violations, cross-language score comparisons remain difficult to interpret.
4. **Reduced Instance Counts for Key Subsets:** While the full dataset contains 494 base instances, downstream tasks use smaller subsets (e.g., only 44 C++ and 77 Java instances for test generation due to resilience against bad-patch generation, Section 3.2.2). These limited samples may constrain statistical power and limit the generalizability of findings for non-Python languages.

### Novelty & Significance
The benchmark is highly novel in its synthetic task augmentation pipeline, which transforms static repository data into interactive, multi-stage software engineering workflows. The methodological emphasis on adversarial test validation (via bad patches) and structured review feedback represents a meaningful step toward evaluating agent reasoning, not just code synthesis. For the ICLR community, OmniCode offers a structured, reproducible testbed to probe generalization, failure modes, and tool-use integration in LLM agents. If accompanied by clear documentation and public release, it will likely serve as a valuable catalyst for advancing robust, human-aligned AI developers.

### Suggestions for Improvement
1. **Incorporate Human Audits for Generated Reviews:** Validate a representative subset of LLM-generated code reviews for accuracy, actionability, and tone. Report human-LLM agreement scores or filter out low-quality reviews to ensure the review-response task tests genuine code comprehension rather than noise resilience.
2. **Clarify Model Versions & Expand Baselines:** Correct and explicitly specify all LLM API versions, temperatures, and cost configurations used in experiments. Include at least 1-2 additional state-of-the-art agent frameworks or specialized test-generation baselines to better contextualize OmniCode's discriminative power.
3. **Formalize and Benchmark the Style Metric:** Provide a rigorous mathematical definition of the style-fixing score, justify threshold selections per language, and compare results against a simple baseline (e.g., regex-based or AST-driven auto-formatters) to isolate genuine agent reasoning from mechanical pattern replacement.
4. **Publish Comprehensive Release Package & Subset Transparency:** Explicitly tabulate instance counts per task/language in the main text and commit to open-sourcing the dataset, Docker environment specifications, random seeds, and exact prompt templates. Clarify the criteria used to filter resilient instances for Java/C++ test generation to ensure reproducibility of the subset selection process.

# Spark Finder Review
## How to Improve This Paper

### Missing Experiments (top 3-5 only)
1. Add empirical data leakage/contamination checks (e.g., training cutoff dates, git commit timestamps vs. model pretraining data) because the core claim that OmniCode "avoids data leakage" is unverified and undermines the credibility of all reported performance scores.
2. Add a strict ablation comparing `Issue + Bad Patch` vs. `Issue + Bad Patch + Review` because providing a partial patch inherently narrows the search space; without this control, the claim that natural-language reviews uniquely improve agent performance is confounded and unconvincing.
3. Add direct LLM prompt baselines (zero/few-shot without agentic loops or tools) for all four tasks because evaluating only complex agent frameworks makes it impossible to disentangle whether performance gaps stem from model reasoning capability or scaffold engineering, which is essential for benchmarking claim validity.

### Deeper Analysis Needed (top 3-5 only)
1. Provide human-verified quality and plausibility scores for a stratified sample of the LLM-generated bad patches because the Test Generation metric entirely hinges on these patches acting as meaningful distractors; trivial or nonsensical bad patches will artificially deflate pass rates.
2. Clarify the exact pass thresholds and normalization procedure for the Style-Fixing score, and report results stratified by linter rule severity because the current metric arbitrarily weights error deletion against error introduction, making cross-language and cross-agent comparisons mathematically ambiguous.
3. Replace the N=4 Pearson correlation analysis with per-instance difficulty stratification or bootstrap resampling because computing correlation coefficients across just four model data points is statistically underpowered and cannot legitimately support claims about task relationships or framing effectiveness.

### Visualizations & Case Studies
1. Include concrete examples of "false positive" tests where agent-generated tests pass all bad patches, detailing why they failed the robustness check, to reveal whether the metric successfully filters out superficial tests or is overly brittle.
2. Visualize before/after code diffs for high-scoring vs. low-scoring Style Fixing tasks, specifically highlighting whether agents fix violations through semantic refactoring or simply delete/comment out flagged code to exploit the metric.
3. Provide representative failure cases for Review-Response where the LLM-generated review accurately identifies the bug but the agent still fails, isolating whether failures stem from poor review quality or the agent’s inability to synthesize feedback.

### Obvious Next Steps
1. Detail the manual validation protocol, including annotator guidelines, number of validators per instance, and inter-annotator agreement scores; without this, the claim that all 1,794 tasks are "manually validated to eliminate ill-defined problems" is not reproducible or trustworthy.
2. Justify the dataset reduction for Test Generation (dropping 61 Java and 68 C++ base instances) and analyze whether discarded instances systematically differ in complexity, otherwise the reported multi-language performance gaps are heavily biased by selective sampling.
3. Report computational efficiency metrics (API cost, token usage, wall-clock time, and iteration counts per successful task) because ICLR standards for agent benchmarks require assessing cost-effectiveness alongside pure capability, especially when arguing that agentic loops outperform pipeline approaches.

# Final Consolidated Review
## Summary
OmniCode introduces a multi-task, multi-language benchmark (1,794 instances across Python, Java, and C++) designed to evaluate LLM-powered software engineering agents beyond isolated bug-fixing. By synthetically augmenting manually curated GitHub pull requests with "bad patches," LLM-generated code reviews, and linter-style violations, the authors create a pipeline that transforms static repository data into interactive evaluation tasks. Experiments with SWE-Agent and Aider reveal significant capability gaps across tasks, particularly in robust test generation and cross-language reasoning, while providing diagnostic insights into agent failure modes.

## Strengths
- **Rigorous test generation evaluation:** The requirement that generated tests must pass the ground-truth patch *and* fail a set of semantically plausible but incorrect "bad patches" (Sec 3.2.2) significantly raises the evaluation bar. Section 5.5 empirically validates this design, showing that traditional gold-only metrics dramatically overstate test quality (~2-6x inflation), confirming that Omnicode's adversarial validation better captures true semantic understanding.
- **Task diversity and synthetic augmentation framework:** Expanding evaluation to four distinct software engineering workflow stages (bug fixing, test generation, review response, style fixing) across three major languages addresses a critical gap in Python-centric benchmarks. The paper provides a clear, reproducible recipe for bootstrapping static PR data into interactive tasks using language-specific tools and LLM perturbations, creating a template for future benchmark construction.
- **Diagnostic depth beyond aggregate scores:** The empirical analysis uncovers actionable behavioral patterns, such as the strong correlation between bug-fixing and review-response capabilities, the "explosive" patch complexity phenomenon on unresolved instances (agents produce sprawling, ineffective refactors when stuck), and clear performance divergences between agentic (SWE-Agent) and pipeline-based (Aider) architectures in iterative, compile-run heavy languages like C++.

## Weaknesses
- **LLM-generated reviews lack validation, risking prompt confounding:** The code review task (Sec 3.2.3) relies entirely on a single LLM (Gemini 2.0 Flash) to generate feedback from bad patches, with no human auditing or automated quality filtering. If reviews are vague, hallucinate constraints, or provide overly directive hints, agent failures conflate poor code comprehension with noisy or misleading instructions, undermining the validity of this task category as a measure of genuine agent reasoning.
- **Distributional coupling in bad patches and systematic subset bias:** The test generation metric's difficulty is intrinsically coupled to the failure modes of specific, now-dated base models (Agentless variants + Gemini perturbations). As noted in Sec 3.2.2, many Java and C++ instances were dropped because they proved "resilient" to bad patch generation, reducing those subsets to 77 and 44 instances respectively. This selective filtering likely biases the multi-language evaluation toward structurally simpler or functionally narrower repositories, limiting cross-language generalizability and making benchmark longevity vulnerable to shifts in how agents fail.
- **Opacity and mathematical ambiguity in the style-fixing metric:** The proposed score `(resolved - new) / original` treats all linter violations equally, ignores severity weighting, and heavily penalizes agents that attempt broader refactors triggering new warnings. The paper mentions "aggressively pruning overly zealous rules" (Sec 3.2.4) but does not specify the pruning criteria or final linter configurations in the main text. This lack of transparency makes it difficult to disentangle genuine agent style reasoning from tool noise or arbitrary penalty structures, weakening cross-language comparisons.

## Nice-to-Haves
- **Statistical variance reporting:** Results are presented as single-point estimates. While understandable given the high cost of repository-level agent runs, reporting multiple seeds or bootstrapped confidence intervals would strengthen claims about marginal performance gaps (e.g., Table 3's 14.0% vs 9.4%).
- **Statistical caution for correlation analysis:** Per-language Pearson correlations are computed across only 4 model data points. These should be framed as descriptive trends rather than statistically robust claims, or supplemented with per-instance difficulty stratification.
- **Non-agentic baselines:** Including a zero/few-shot prompt-to-patch baseline without tool loops or execution feedback would help isolate the contribution of the agent scaffold versus raw model capability, though this is not strictly required for the benchmark's stated focus.

## Novel Insights
The benchmark surfaces a distinct failure mode in autonomous coding agents: when agents cannot identify a precise fix, they frequently resort to "explosive" complexity, generating sprawling, multi-file refactors that drastically exceed human patch size (e.g., GPT-5-mini unresolved Python patches averaging a complexity score of 390 vs. gold 7.07). Successful resolutions, conversely, are often cleaner than the original human patches. Additionally, the style-fixing results reveal a sharp capability boundary: agents reliably fix local, syntactic violations but diverge significantly on tasks requiring design-level judgment or cross-file synthesis (e.g., converting to utility classes, safe renames). Together, these patterns suggest current LLM agents excel at deterministic, pattern-matched edits but lack robust intent inference and non-local semantic reasoning.

## Suggestions
- **Audit or filter LLM-generated reviews:** Manually validate a stratified sample of generated reviews for clarity and actionability, or implement automated constraints (e.g., requiring specific line references and prohibiting direct code provision) to ensure the task measures agent synthesis rather than noise tolerance.
- **Clarify bad patch generation and subset filtering:** Explicitly document the criteria used to retain/drop Java and C++ test-generation instances and analyze whether discarded instances differ systematically in complexity or repository type. Consider diversifying bad patches via rule-based mutation or historical bug databases to reduce coupling to specific LLM failure modes.
- **Formalize the style metric and release configurations:** Provide the exact linter rulesets, pruning methodology, and severity weights used. Reframe or justify the scoring formula against standard precision/recall or cost-benefit thresholds, and report results stratified by rule severity to clarify whether agents struggle with tool interaction or actual style reasoning.
