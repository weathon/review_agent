# Harsh Critic Review
## Section-by-Section Critical Review

### Title & Abstract
- **Does the title accurately reflect the contribution?** Yes. "Dynamic Interactions" and "Re-imagining Text-to-SQL Evaluation" accurately capture the benchmark's focus on multi-turn, stateful, and agentic database querying rather than static single-turn generation.
- **Does the abstract clearly state the problem, method, and key results?** Broadly yes. It identifies the gap (static transcripts, SELECT-only limits), introduces the environment, two evaluation settings, and the dataset split. However, there is a significant numerical ambiguity: the abstract reports GPT-5 achieving **8.67%** in *c*-Interact and **17.00%** in *a*-Interact, while Table 2 reports **14.50%** and **29.17%** respectively for the overall priority sub-task success. The abstract does not clarify whether these percentages represent *end-to-end joint task completion* (both sub-tasks solved) versus the *per-subtask SR* reported later. This creates immediate confusion about the true performance floor.
- **Are any claims in the abstract unsupported by the paper?** The claim that the benchmark "restores missing realism" is slightly rhetorical given that ambiguities are synthetically injected and user responses are simulated via a function-driven LLM. While well-justified in the method, realism in benchmarks is inherently bounded by simulator design. The abstract should temper this phrasing or explicitly note the simulated nature of the interaction environment upfront.

### Introduction & Motivation
- **Is the problem well-motivated? Is the gap in prior work clearly identified?** Yes. The critique of static conversation transcripts (e.g., CoSQL, SParC) and SELECT-only limitations is well-founded and aligns with known production challenges where user intent evolves and requires clarification/error recovery. The gap is clearly framed.
- **Are the contributions clearly stated and accurate?** Yes. The three contributions (interactive environment/simulator, two evaluation settings, comprehensive task suite) are explicitly listed and match the subsequent sections.
- **Does the introduction over-claim or under-sell?** It slightly over-claims by positioning BIRD-INTERACT as a direct proxy for "production-grade database assistant" without acknowledging that injected ambiguities may follow a different distribution than organic human confusion. Real users rarely present ambiguities that cleanly map to hidden SQL fragments or DAG-structured knowledge chains. The introduction should more carefully delineate between *simulated* interaction realism and *empirical* realism.

### Method / Approach
- **Is the method clearly described and reproducible?** The construction pipeline is well-documented across Sections 3 and Appendices C, H, and N. The annotation protocol, ambiguity taxonomy, and two-stage simulator design are transparent. The use of LIVESQLBENCH as a foundation ensures structural reproducibility.
- **Are key assumptions stated and justified?** The paper assumes that resolving injected ambiguities requires interaction. This is validated by the "necessity of clarification" checklist (App H.1), but it assumes human users will ask the same clarification questions that the system generates. The assumption that a *function-driven simulator* adequately approximates human user behavior is partially validated via correlation studies (Sec 6, Table 3), but the underlying reliance on GT SQL for answer generation (LOC/AMB actions) means the simulator is effectively a *teaching oracle* rather than a true user proxy. This is a valid evaluation design, but should be explicitly framed as such.
- **Are there logical gaps in the derivation or reasoning?** The `LOC()` action relies on AST-based retrieval to map system questions to GT SQL fragments (App N). While structurally sound, AST matching assumes syntactic correspondence between the user's natural language question and specific SQL clauses. It does not explicitly address semantic equivalence where valid clarifications might map to logically equivalent but syntactically distinct SQL patterns. The robustness of the parser to diverse phrasings is not ablated.
- **Are there edge cases or failure modes not discussed?** The `a`-Interact setting terminates if the priority sub-task fails (Sec 4.1), which prevents evaluation of error recovery or alternative strategy pivoting when a model's initial approach is fundamentally misaligned. This is a deliberate design choice but limits the "agentic" evaluation to early-commit success. Additionally, the budget constraint in `a`-Interact heavily penalizes exploration (execute=1, ask=2, submit=3), which may artificially suppress performance by incentivizing premature submission rather than thorough schema/knowledge investigation.

### Experiments & Results
- **Do the experiments actually test the paper's claims?** Yes. The main results, interaction scaling (Fig 4), memory grafting (Fig 5), and action distribution (Fig 12/13) directly test interaction effectiveness, communication bottlenecks, and strategic tool use.
- **Are baselines appropriate and fairly compared?** The paper evaluates 7 frontier LLMs using standard decoding (`temp=0`) and default reasoning settings. However, it lacks comparisons with *specialized text-to-SQL agents* (e.g., MAC-SQL, DTS-SQL, Xiyan-SQL) or even strong few-shot prompt engineering baselines. Given the benchmark's focus on pushing the state of the art, evaluating only out-of-the-box API models may understate achievable performance and limits the paper's utility for method development.
- **Are there missing ablations that would materially change conclusions?** An ablation comparing performance with vs. without the hierarchical knowledge base (HKB) would isolate whether low success rates stem from SQL generation weaknesses or knowledge retrieval failures. Additionally, varying the budget constraint more granularly in `a`-Interact (beyond the reported ITS patience sweep) would clarify whether the "submit-heavy" or "execute-heavy" behaviors are optimal under different compute regimes.
- **Are error bars / statistical significance reported?** Experiments are run as single deterministic passes due to API costs (Sec 5, App I.3). While deterministic decoding mitigates output variance, it does not account for prompt sensitivity, system message framing, or minor API version shifts. Reporting at least 3 seeds for open models or acknowledging the lack of variance metrics is necessary for ICLR's empirical rigor standards.
- **Do the results support the claims made, or are they cherry-picked?** Results consistently show low SR across models, supporting the difficulty claim. The memory grafting and action distribution analyses are honest and not cherry-picked. However, the divergence between abstract metrics and Table 2 (as noted above) needs reconciliation to ensure claims are accurately calibrated.
- **Are datasets and evaluation metrics appropriate?** The SR and Normalized Reward metrics are well-defined and aligned with the task structure. Executable test cases for BI (soft EM) and DM (custom assertions) are appropriate. The 70/30 weighting for priority vs. follow-up rewards is reasonable given the sequential dependency.

### Writing & Clarity
- **Are there sections that are confusing or poorly explained?** Section 2's formalization of the interaction turn (Eq 1) is intuitive but mixes history concatenation notation with reward definitions that appear later. The transition from `c`-Interact (clarification-turn budget) to `a`-Interact (action-cost budget) is clear, but the rationale for the specific cost assignments (0.5, 1, 2, 3) in Table 9 is briefly explained as empirical; a brief sensitivity analysis or citation for these weights would strengthen reproducibility.
- **Are figures and tables clear and informative?** Table 2's column headers ("Priority Question", "Follow Ups", "Overall") are slightly ambiguous regarding whether they reflect per-subtask or cumulative task success. Figure 1 suffers from severe parser artifacts in the text extraction but appears structurally sound in design. Figures 4 and 5 are clear and effectively communicate interaction scaling and memory grafting effects.

### Limitations & Broader Impact
- **Do the authors acknowledge the key limitations?** Yes. The authors note the focus on text-to-SQL (App A), the synthetic nature of ambiguity injection, and the stress-mode budget constraints (Sec 8). They also acknowledge reliance on LLM simulators despite the two-stage guard.
- **Are there fundamental limitations they missed?** The sequential dependency design (failure in sub-task 1 terminates the session) inherently measures *first-attempt disambiguation accuracy* more than *long-horizon interactive recovery*. Additionally, the benchmark does not address multi-user or concurrent query interference, which is a real-world DB production factor. The simulator's reliance on GT SQL for generating clarifications means it cannot evaluate scenarios where the *user's initial goal is genuinely ill-posed* and requires iterative goal refinement rather than disambiguation of a hidden correct SQL.
- **Are there failure modes or negative societal impacts not discussed?** Failure modes include models learning to "game" the simulator's AST retrieval or LLM parser through prompt-specific phrasing rather than genuine reasoning. The paper briefly covers ethics (no human subjects), but does not discuss the risk of benchmark leakage if future models train on the released interaction trajectories, which could artificially inflate future scores.

### Overall Assessment
BIRD-INTERACT presents a timely and well-structured benchmark that meaningfully advances text-to-SQL evaluation by moving beyond static, single-turn, read-only paradigms. The two-stage function-driven user simulator is a strong methodological contribution that effectively mitigates ground-truth leakage, and the dual evaluation settings (`c`-Interact and `a`-Interact) provide valuable insights into how frontier models balance exploration, clarification, and execution under resource constraints. The empirical analyses (Interaction Test-Time Scaling, memory grafting, action distribution) are insightful and support the paper's core thesis: strategic interaction capability, not just SQL generation, is the primary bottleneck. However, several issues must be addressed before publication. First, the numerical discrepancy between the abstract (8.67%/17.00%) and Table 2 (14.50%/29.17%) must be clarified (likely per-subtask vs. joint task success). Second, the absence of specialized multi-turn text-to-SQL agent baselines limits the benchmark's positioning as a catalyst for method development. Third, single-run evaluation without variance reporting falls short of ICLR's empirical rigor standards, even with deterministic decoding. Finally, the authors should more explicitly frame the simulator as a *teaching oracle* rather than a human proxy, and discuss how the sequential sub-task dependency shapes the evaluation's scope. With these clarifications, minor ablations, and broader baseline comparisons, this work represents a strong, valuable contribution to the interactive database agent literature.

# Neutral Reviewer
## Balanced Review

### Summary
The paper introduces BIRD-INTERACT, a dynamic multi-turn benchmark designed to evaluate LLMs in realistic, interactive text-to-SQL scenarios. Unlike prior datasets that rely on static conversation histories or read-only queries, BIRD-INTERACT integrates a function-driven user simulator, hierarchical knowledge bases, and full CRUD operations, evaluated under both protocol-guided (_c_-Interact) and autonomous agentic (_a_-Interact) settings. The authors demonstrate that frontier models struggle significantly with the benchmark, providing diagnostic analyses on interaction strategies, memory grafting, and interaction test-time scaling to highlight the critical gap between SQL generation and strategic communication.

### Strengths
1. **High Ecological Validity & Comprehensive Scope:** The benchmark successfully transitions text-to-SQL evaluation from idealized single-turn prompts to realistic, stateful, multi-turn interactions. The inclusion of full CRUD operations (not just SELECT), dynamic ambiguity injection, and hierarchical knowledge dependencies accurately reflects production database assistance workflows.
2. **Robust Function-Driven User Simulator:** Addressing a known failure mode in LLM-as-user simulators (ground-truth leakage), the two-stage function-driven architecture (mapping requests to constrained actions like AMB, LOC, UNA) significantly improves reliability. The dedicated UserSim-Guard evaluation rigorously quantifies this, showing a dramatic drop in failure rates for unanswerable queries compared to baselines.
3. **Thorough Annotation & Quality Control:** The pipeline involves a rigorous, multi-stage expert annotation process with high inter-annotator agreement (93.33%+) and independent human quality validation (97.3% acceptance). This ensures high data integrity and strong reproducibility guarantees.
4. **Insightful Diagnostic Analyses:** Beyond standard accuracy metrics, the paper introduces valuable analytical dimensions like Interaction Test-Time Scaling (ITS), action distribution heatmaps, and Memory Grafting. These effectively diagnose model weaknesses, such as the tendency of weaker systems to over-rely on brute-force execution rather than strategic environment exploration.
5. **Budget-Constrained Evaluation Framework:** Implementing explicit action costs and adaptive budget constraints realistically mirrors computational and user-patience limits, pushing evaluated systems toward efficient, cost-aware interaction strategies rather than unbounded trial-and-error.

### Weaknesses
1. **Single-Run Evaluations & Lack of Statistical Significance Testing:** Due to high API costs, all frontier models are evaluated in a single run. While deterministic decoding is used, LLM outputs remain inherently variable. The absence of confidence intervals or bootstrapping around Success Rates makes it difficult to determine whether minor performance gaps between top-performing models (e.g., 1-2% differences) are statistically meaningful.
2. **Heavy Reliance on Closed-Source/Proprietary Models:** The empirical evaluation focuses almost exclusively on expensive, closed-source APIs (GPT-5, Claude-Sonnet-4, Gemini-2.5-Pro, etc.). For an ICLR submission, the lack of comprehensive evaluations across strong open-weight models (e.g., Llama-3.3-70B, Qwen-Coder series) limits the broader academic community's ability to access, fine-tune, and iterate on the benchmark.
3. **Potential Prompt Brittleness:** The system prompts for both _c_-Interact and _a_-Interact (Appendix R) are highly engineered, relying on strict XML-like tags, repeated warnings, and explicit formatting constraints. The benchmark's reported performance may be heavily sensitive to prompt design and cost-scheme tuning, potentially conflating prompt optimization with underlying model reasoning.
4. **Underspecified "Memory Grafting" Methodology:** The Memory Grafting experiment (Figure 5) is used to prove GPT-5's generation capabilities versus poor communication skills, but the exact methodology is opaque. It is unclear how interaction trajectories from other models are aligned, whether the budget is adjusted for the grafted history, or if this setup inadvertently breaks the dynamic constraints of the original evaluation.
5. **Lack of Training/Algorithmic Baselines:** While excellent as an evaluation suite, the paper stops short of demonstrating how models can be trained to improve on BIRD-INTERACT. There are no fine-tuning (SFT/RL) baselines or reinforcement learning setups provided, which leaves the "path to improvement" largely theoretical for researchers adopting the benchmark.

### Novelty & Significance
**Novelty:** The work makes substantial methodological contributions to interactive database reasoning. By moving away from static dialogue transcripts to live, stateful environment interaction, and coupling this with a robust function-driven simulator and a dual-mode (agentic vs. conversational) evaluation framework, it addresses a critical gap in contemporary NLIDB research.
**Significance:** BIRD-INTERACT will likely serve as a vital milestone for the LLM agent and database communities. It accurately diagnoses the current bottleneck in text-to-SQL systems: not SQL syntax generation, but strategic communication, ambiguity resolution, and efficient environment exploration. The budget-aware design and detailed action analyses provide a strong, production-relevant blueprint for developing genuinely useful autonomous database assistants.

### Suggestions for Improvement
1. **Incorporate Statistical Rigor:** Perform bootstrap resampling over the task set to report 95% confidence intervals for Success Rate and Normalized Reward. This is essential for distinguishing meaningful architectural or scaling improvements from evaluation noise.
2. **Benchmark Open-Weight Models:** Include a suite of strong, publicly accessible open-weight models in the main experimental tables. This aligns with ICLR's emphasis on open science and will make the benchmark immediately usable for academic researchers without API budgets.
3. **Formalize Memory Grafting & Provide Ablations:** Clearly detail the Memory Grafting protocol (e.g., prompt injection method, budget handling). Additionally, include a brief sensitivity analysis demonstrating how results change with different base budgets (`B_base`) or prompt variations to prove the benchmark measures inherent capability rather than prompt tuning.
4. **Provide a Training Baseline or RL Setup:** To maximize impact, provide a starter repository with at least one SFT or RL fine-tuning loop (e.g., using GRPO or PPO on the interaction logs) to show researchers how the benchmark can actively drive algorithmic improvement, not just static evaluation.
5. **Release a Streamlined Evaluation Harness:** Ensure the accompanying code release includes a plug-and-play evaluation script, clear instructions for custom agent integration, and a minimal baseline ReAct agent. This will drastically lower the barrier to entry and accelerate community adoption.

# Spark Finder Review
## How to Improve This Paper

### Missing Experiments (top 3-5 only)
1. Benchmark against specialized interactive/agent text-to-SQL frameworks (e.g., adapted MAC-SQL, Reflexion, or DAIL-SQL) rather than only base LLMs. Without these, claims about benchmark difficulty are confounded with weak prompting baselines.
2. Include an oracle upper-bound experiment where all injected ambiguities are perfectly resolved before generation. This is required to isolate whether the ~17% success rate stems from interaction strategy failures or fundamental SQL generation limits.
3. Report results across multiple seeds with standard deviations instead of single deterministic runs. Interactive exploration paths are highly sensitive to decoding variance, and single-run results lack the statistical rigor expected at ICLR.
4. Ablate the impact of the two-stage user simulator design on the main benchmark results. Run a subset of tasks with a standard LLM simulator that permits mild ground-truth leakage to prove your function-driven design actually changes evaluation outcomes rather than just adding structural complexity.

### Deeper Analysis Needed (top 3-5 only)
1. Disaggregate failure modes into distinct categories: inability to detect ambiguities, failure to formulate targeted clarification questions, context forgetting across turns, and post-clarification SQL errors. The current high-level error sampling does not pinpoint the bottleneck for method development.
2. Quantify actual state utilization for follow-up subtasks. Provide a metric measuring how often models query newly created or modified database objects versus relying solely on static schema knowledge, directly verifying the "state dependency" claim.
3. Establish causality between action distribution balance and success rather than relying on correlation. Intervene by constraining weaker models to use balanced action mixes or forcing exploration phases; without this, the "balanced strategy" finding may simply be a proxy for overall model capability.

### Visualizations & Case Studies
1. Provide side-by-side interaction trajectories (transcript + action timeline) for top-performing vs. failing models in both settings. This directly reveals whether a-Interact agents fail due to premature SQL submission, redundant environment probing, or poor clarification phrasing.
2. Visualize a Sankey diagram mapping ambiguity types -> model's resolution action -> outcome (success/failure/timeout). This exposes systematic blind spots in how current architectures route their reasoning to resolve specific knowledge or query ambiguities.

### Obvious Next Steps
1. Develop and release a purpose-built agentic scaffold optimized for the BIRD-INTERACT action space as a primary baseline. The community needs a strong, reproducible starting point that demonstrates how to effectively leverage the benchmark's environment, not just raw API responses.
2. Conduct a rigorous sensitivity analysis on budget hyperparameters (`λ_pat`, `B_base`, and action costs). Demonstrate that the low success rates persist across relaxed budgets to prove the benchmark measures strategic capability rather than arbitrary resource starvation.
3. Scale the human-simulator alignment study beyond 100 tasks. The current sample size is too small to confidently claim the function-driven simulator generalizes to human behavior patterns across diverse BI and DM domains at the scale of the full dataset.

# Final Consolidated Review
## Summary
The paper introduces BIRD-INTERACT, a dynamic multi-turn benchmark that shifts text-to-SQL evaluation from static, single-turn generation to live, stateful interaction. It features a function-driven user simulator designed to prevent ground-truth leakage, dual evaluation modes (protocol-guided *c*-Interact and autonomous *a*-Interact), and tasks spanning full CRUD operations with injected ambiguities and sequential state dependencies. Empirical results across frontier models reveal low success rates, supported by diagnostic analyses demonstrating that strategic communication and efficient action sequencing, rather than raw SQL generation capability, are the primary bottlenecks.

## Strengths
- **Principled shift to live, stateful interaction:** The benchmark accurately models production database workflows by enforcing sequential sub-task dependency, injecting multi-source ambiguities (query, knowledge chain, environment), and supporting full CRUD operations. This design forces systems to actively disambiguate and adapt rather than rely on static context windows.
- **Robust, validated function-driven user simulator:** The two-stage action-classification pipeline (AMB/LOC/UNA) directly addresses the pervasive ground-truth leakage problem in LLM-as-user simulators. Rigorous validation on UserSim-Guard and strong human-alignment correlation (Pearson up to 0.84) substantiate claims of controllability without sacrificing interaction realism.
- **Actionable diagnostic framework beyond accuracy:** The introduction of Interaction Test-Time Scaling (ITS), action distribution heatmaps, and memory grafting provides concrete, interpretable insights into *why* models fail. The finding that performance scales monotonically with interaction turns, and that weaker models misallocate budget toward brute-force execution, offers clear direction for future agent scaffold design.

## Weaknesses
- **Evaluation limited to out-of-the-box frontier LLMs:** The paper benchmarks only raw API models under default settings without adapting specialized text-to-SQL frameworks (e.g., MAC-SQL, DTS-SQL) or even strong few-shot agentic scaffolds. This conflates baseline prompting weakness with inherent benchmark difficulty and limits the paper's immediate utility for driving method development in the agent community.
- **Lack of variance reporting and budget sensitivity analysis:** All experiments are single deterministic runs. While `temperature=0` mitigates sampling variance, it does not account for API response drift, prompt framing sensitivity, or stochastic decoding in open models. Furthermore, the strong conclusions about action-cost trade-offs (e.g., why models over-invest in `submit` vs `ask`) are drawn from a single budget configuration (`λ_pat=3`, `B_base=6`). Without sensitivity analysis or bootstrapped confidence intervals, it remains unclear whether the observed interaction patterns reflect inherent model behavior or arbitrary hyperparameter choices.

## Nice-to-Haves
- Clarify that the abstract's reported success rates (8.67% / 17.00%) correspond to end-to-end joint task completion (equivalent to follow-up sub-task SR due to sequential termination), while Table 2 reports per-subtask rates, preventing initial reader confusion.
- Include an "oracle" upper-bound experiment where ambiguities are pre-resolved, isolating whether low success rates stem from interaction strategy failures or fundamental SQL generation/synthesis limits.
- Formalize the exact prompt injection and budget-handling protocol for the memory grafting experiment in the main text or appendix for full reproducibility.
- Release a lightweight, plug-and-play evaluation harness alongside the benchmark, ideally accompanied by a strong open-weight ReAct/agent baseline to accelerate community adoption without API dependencies.

## Novel Insights
The paper successfully reframes the text-to-SQL bottleneck from syntactic parsing to strategic human-AI collaboration. The empirical evidence strongly suggests that contemporary frontier models possess adequate SQL generation capacity but lack calibrated metacognition for information gathering: they frequently skip necessary clarifications, misprioritize expensive trial-and-error execution over low-cost schema/knowledge probing, and struggle to maintain state across evolving database contexts. The monotonic scaling of performance with additional interaction turns further confirms that current models are not fundamentally incapable, but rather lack learned policies for efficient resource allocation during uncertain, multi-step reasoning tasks.

## Suggestions
1. **Add specialized agent baselines:** Benchmark at least 2-3 adapted text-to-SQL agentic methods or scaffolded pipelines (even open-weight) to establish a realistic performance floor/ceiling and demonstrate how the benchmark can track methodological progress beyond base model scaling.
2. **Report budget/prompt robustness metrics:** Provide a brief sensitivity sweep across at least two budget configurations (e.g., relaxed `λ_pat=5/7`) and note the impact on action distributions. If feasible, run 3 seeds on the lowest-cost open models with standard deviations or report bootstrapped 95% CIs across the task set to strengthen claims about model ranking and strategic patterns.
3. **Explicitly define end-to-end vs. per-subtask metrics:** Add a clarifying sentence in Section 2 or 4.1 stating that due to sequential termination, joint task success is numerically identical to the follow-up sub-task SR, justifying the abstract's reporting choice.
